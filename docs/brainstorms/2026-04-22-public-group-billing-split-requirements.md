# Public Group Billing Split Requirements

## Problem Frame

当前系统里的 `group` 同时承担了用户可见分组、模型展示、渠道选择、分组倍率等职责。现有能力足以支持“一个公开组对应一个固定倍率”，但不足以支持下面这类场景：

- 用户侧仍然只看到少量公开组，例如：
  - `ask-第三方渠道`
  - `ask-CC专用`
- 同一个公开组下会展示其上游渠道实际支持的所有模型
- 同一个模型可以同时出现在多个公开组
- 真实计费又需要按命中的**渠道标签**区分，例如：
  - `GPT`
  - `Claude 第三方`
  - `Claude Code`

如果继续只按公开组倍率计算，会出现前台价格、实际扣费、日志解释三者不一致的问题。这个需求的目标不是改变用户看到的公开组，而是在不破坏现有公开组体验的前提下，让模型展示、渠道路由、计费倍率和日志文案保持同一套口径。

## Requirements

**公开组与模型展示**
- R1. 系统必须保留现有公开组概念，用户继续按公开组使用系统，不需要看到内部计费组名称。
- R2. 每个公开组必须展示该公开组背后可用渠道实际支持的所有模型，而不是手工维护一份单独的模型列表。
- R3. 同一个模型允许同时出现在多个公开组中；不同公开组下该模型的最终价格可以不同。

**计费归属解析**
- R4. 系统必须支持按“公开组 + 渠道标签”定义真实计费归属规则，并以该规则作为倍率计算来源。
- R5. 系统必须支持按“公开组 + 模型”进行可选覆盖，用于在自动推导存在歧义时，强制指定该模型在该公开组下命中的渠道标签。
- R6. 当未配置模型覆盖时，系统应根据当前公开组下可用渠道的模型支持情况与渠道标签，自动推导该模型命中的渠道标签。
- R7. 对于同一个“公开组 + 模型”，系统必须稳定解析为唯一渠道标签；不允许同一请求在多个渠道标签之间漂移。
- R8. 当未命中渠道标签，或命中了渠道标签但未配置对应的真实计费规则时，系统必须回退为公开组默认倍率计算。

**请求链路与运行时行为**
- R9. 渠道标签与真实计费归属的解析必须发生在认证得到公开组之后、已拿到模型名之后、正式选择渠道之前。
- R10. 正式选渠道时，系统必须在“公开组 + 模型”基础上进一步限定到解析出的唯一渠道标签范围内，避免跨标签选中错误渠道。
- R11. 实际扣费必须按真实计费归属对应的倍率计算，而不是固定按公开组倍率计算。
- R12. 前台价格展示必须与运行时真实扣费保持同一口径；同一公开组下不同模型可以因命中不同渠道标签而展示不同价格。

**日志与文案**
- R13. 管理员侧日志详情必须额外展示：
  - 用户选择分组
  - 命中渠道标签
  - 实际计费倍率
- R14. 用户侧日志详情必须展示：
  - 计费归属（显示命中的渠道标签名称）
  - 计费倍率
- R15. 当请求走的是公开组默认倍率回退逻辑时，管理员侧和用户侧日志都必须能够明确表达这是默认规则，而不是标签命中结果。
- R16. 计费过程文案中的倍率说明必须反映真实使用的倍率来源，不能继续默认写成公开组倍率。

## Success Criteria

- 同一请求在前台价格、实际扣费、管理员日志、用户日志中的倍率口径一致。
- 用户仍然只感知公开组，不需要理解内部计费组。
- 管理员可以从日志详情中直接看出这次请求命中了哪个渠道标签、按哪个倍率结算。
- 对于未配置规则的情况，系统能稳定回退到公开组默认倍率，不会出现空值或错误倍率。
- 同一个“公开组 + 模型”在运行时稳定命中单一渠道标签，不会因渠道切换导致计费解释变化。

## Scope Boundaries

- 不要求在本次需求里重构整个 `group` 体系的历史职责，只要求补足公开组下按渠道标签分流计费的能力。
- 不要求用户侧暴露内部计费组名称。
- 不要求引入新的独立“计费标签”字段；本次直接复用现有渠道标签。
- 不要求通过 Nginx 或其他前置网关来解决此问题；该需求以内置代码能力为主。
- 不要求为每个模型手工维护完整静态映射表；系统应以自动推导为主、人工覆盖为辅。

## Key Decisions

- 保留公开组不变：用户体验不改，避免把内部计费结构暴露给用户。
- 复用渠道标签作为计费归属语义：避免引入新的配置概念，降低运营维护成本。
- 以“自动推导 + 人工覆盖”作为关系来源：默认依赖渠道能力和标签自动解析，仅在歧义时手工钉死。
- 将关系解析放在“认证后、选渠道前”：这是同时满足路由、计费、展示、日志统一口径的最小改动位置。
- 用户侧文案使用“计费归属”显示渠道标签名称：比展示内部计费组名称更直观。

## Dependencies / Assumptions

- 当前渠道标签值可以较稳定地表达业务语义，例如 `GPT`、`Claude 第三方`、`Claude Code`。
- 业务上可以保证同一个“公开组 + 模型”最终只会命中一种渠道标签。
- 现有渠道与 ability 数据足以支持按公开组、模型、标签进行自动推导。
